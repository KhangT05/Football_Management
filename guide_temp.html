1. Kiến trúc tổng quan

Traffic vào qua 1 entrypoint Nginx (port 80), route theo path prefix tới 2 service backend:

  • api (Express + TSOA, port 3000 nội bộ) — REST API, Swagger UI, business logic.
  • frontend (Nginx-alpine phục vụ static build của Vite/React, port 80 nội bộ).
  • MySQL— primary datastore (Prisma 7, driver adapter @prisma/adapter-mariadb).
  • Redis — cache / session / rate-limit backing store.

Client → Nginx (80/443)
├─ / → frontend:80 (SPA static, try_files → index.html)
├─ /api/ → api:3000 (REST)
└─ /swagger → api:3000 (Swagger UI)

api → mysql:3306 (Prisma), redis:6379 (cache)

2. Yêu cầu hệ thống

  • Node.js ≥ 20 (khớp base image node:20-alpine / node:22-alpine trong Dockerfile).
  • Docker + Docker Compose v2 (nếu chạy theo compose).
  • MySQL 8.0 hoặc MariaDB tương thích (compose dùng image mysql:8.0, nhưng client driver là @prisma/adapter-mariadb + mariadb package — cần verify tương thích wire-protocol nếu đổi sang MariaDB thật).
  • Redis 7.x.

5. Chạy local dev (không qua Docker)

5.1 Backend (api/)

cd api
cp .env.example .env # điền các key ở mục 4, DATABASE_URL trỏ localhost
npm ci
npx prisma generate # sinh client vào src/generated/prisma
npx prisma migrate dev # apply schema + tạo migration mới nếu schema thay đổi
npm run dev # tsx watch src/app.ts — hot reload

Cần MySQL/MariaDB và Redis chạy sẵn (local install hoặc docker compose up mysql redis -d từ file compose gốc để chỉ start 2 dependency, không start toàn bộ stack).

5.2 Frontend (frontend/)

cd frontend
npm ci
npm run dev # Vite dev server, mặc định port 3001 (server.port trong vite.config.js)

Vite dev proxy /api hiện hardcode target là 1 IP cụ thể trong vite.config.js — cần sửa thành target: process.env.VITE_DEV_API_URL || 'http://localhost:3000' trước khi dùng, nếu không dev sẽ gọi nhầm sang server ngoài thay vì backend local.

6. Chạy bằng Docker Compose (production-like)

cp .env.example .env # điền đủ key mục 4
docker compose up -d --build
docker compose logs -f api # api container tự chạy: prisma migrate deploy && node dist/app.js

Thứ tự khởi động thực tế:

  • mysql, redis start trước (dependency của api).
  • api start sau mysql/redis nhưng compose chỉ dùng depends_on mặc định (chờ container start, không chờ MySQL sẵn sàng nhận connection) → lần đầu deploy trên máy chậm có thể fail ở bước prisma migrate deploy. Nếu gặp lỗi connection refused, chạy lại: docker compose up -d api.
  • nginx start sau cùng, phụ thuộc api + frontend.

Truy cập:

  • Frontend: http://localhost/
  • API: http://localhost/api/...
  • Swagger: http://localhost/swagger

7. Database — Prisma / Migration

  • Generator dùng provider mới prisma-client (Prisma 7), output custom vào api/src/generated/prisma thay vì @prisma/client mặc định — khi import client trong code phải import từ đúng path generated này, không phải từ '@prisma/client'.
  • Dùng driver adapter (@prisma/adapter-mariadb) → khởi tạo PrismaClient phải truyền adapter object, không chỉ đọc DATABASE_URL tự động như Prisma 5/6 mặc định. Kiểm tra src/lib/prisma.ts (hoặc tương đương) có đúng pattern new PrismaClient({ adapter }) chưa.
  • Migration production: Dockerfile CMD chạy prisma migrate deploy mỗi lần container start — an toàn cho 1 replica. Nếu sau này scale api ra nhiều container, tách migrate deploy thành 1 job/step riêng trong CI/CD (chạy 1 lần trước khi rolling update), tránh race điều kiện concurrent migration.
  • Model đáng chú ý cho vận hành: TournamentRule.tiebreaker_order và custom_stages là JSON tự do — validate ở tầng application (Zod), Prisma không enforce schema của JSON field này.

8. Nginx — Routing & khuyến nghị sửa

8.1 Config hiện tại

nginx.conf (edge) route theo 3 location: / → frontend, /swagger và /api/ → api:3000, dùng biến upstream + resolver 127.0.0.11 để nginx resolve lại DNS Docker mỗi 5s (đúng pattern — tránh nginx cache IP cũ khi container api/frontend bị restart).

8.2 Vấn đề thật cần fix trước khi lên production

  • Thiếu location cho WebSocket (/socket.io/). Frontend có dependency socket.io-client — nếu backend dùng socket.io cho realtime (điểm số trực tiếp, thông báo), request /socket.io/ hiện sẽ rơi vào location / (proxy sang frontend, sai target) vì không match /api/. Cần thêm location riêng với proxy_http_version 1.1 và header Upgrade/Connection:

location /socket.io/ {
proxy_pass http://api:3000;
proxy_http_version 1.1;
proxy_set_header Upgrade $http_upgrade;
proxy_set_header Connection "upgrade";
proxy_set_header Host $host;
proxy_read_timeout 3600s;
}

  • 443 được publish trong docker-compose và volume certbot/www được mount, nhưng nginx.conf chỉ có server { listen 80 } — chưa có server block 443 ssl với ssl_certificate/ssl_certificate_key, và compose không có service certbot để renew. SSL hiện chưa thực sự hoạt động dù port đã mở.
  • api container vừa publish ports: "3000:3000" ra host vừa được nginx proxy nội bộ qua /api/ — port 3000 mở public là attack surface thừa nếu client luôn đi qua domain chính. Bỏ ports mapping của api service, chỉ giữ expose nội bộ trong network app_network.

9. Bảo mật / vận hành

Tạo MySQL user riêng cho app (GRANT giới hạn trên đúng 1 database), không dùng root trong DATABASE_URL — hiện compose chỉ có MYSQL_ROOT_PASSWORD, chưa có user ứng dụng.
